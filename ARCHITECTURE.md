## Overview

This document describes the architecture of a **file download microservice** designed to handle variable processing times (10-120 seconds) while maintaining reliability behind reverse proxies like Cloudflare, nginx, or AWS ALB.

---

## Integration Scenario

### Real-World Use Case

This file download service is designed to be **integrated with external applications** (e.g., e-commerce platforms, document management systems, content delivery platforms) to handle complex file processing workflows.

### Problems Encountered

When deployed behind reverse proxies, we face:

1. **â±ï¸ Connection Timeouts**
   - Cloudflare: 100s default timeout
   - nginx: 60s default timeout
   - AWS ALB: 60s default timeout
   - **Result**: Long-running downloads (>60s) get terminated

2. **ğŸ‘¤ Poor User Experience**
   - Users wait 2+ minutes with no feedback
   - No progress indication
   - Connection drops = lost work

3. **ğŸ’» Resource Exhaustion**
   - Holding HTTP connections open for 2+ minutes
   - Server resources tied up waiting
   - Limited concurrent request capacity

4. **ğŸ”„ Retry Storms**
   - Dropped connections trigger client retries
   - Duplicate work and wasted resources
   - Potential data inconsistency

## Summary

### Key Architectural Decisions

1. **Immediate Response Pattern**: `/v1/download/initiate` returns immediately with a `jobId`, avoiding timeout issues
2. **Async Processing**: Long-running operations handled by background workers
3. **Real-time Updates**: Server-Sent Events (SSE) for progress updates
4. **S3-Compatible Storage**: MinIO for self-hosted object storage
5. **Observability**: OpenTelemetry + Sentry for comprehensive monitoring
6. **Security**: Rate limiting, CORS, security headers, input validation

### Benefits

âœ… **No Timeout Issues**: Immediate responses prevent proxy timeouts  
âœ… **Better UX**: Real-time progress updates via SSE
